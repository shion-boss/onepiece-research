# ワンピースカード 戦い方のコツ

> このスキルは **戦略の知見** であり、ルール (engine 実装) ではない。
> ルール厳密性は `onepiece-tcg-rules` skill を参照。
> ここに書いてある「コツ」は AI ヒューリスティックの妥当性チェック、デッキ評価軸の根拠、
> プレイ補助 UI の判断基準として使う。

---

## このスキルの使いどき

| 状況 | 参照箇所 |
|---|---|
| `engine/ai.py` の `choose_action` / `choose_defense` を強化する | §3 アタック / §5 カウンター / §11 AI研究環境 |
| デッキ評価ダッシュボードに新しい軸を追加する | §7 デッキ構築 / §8 環境読み |
| 勝率行列の異常値を解釈する (なぜこのデッキが弱い?) | §1 基本戦略 / §6 盤面管理 |
| 「無意味な攻撃」「過剰防衛」の閾値設計 | §3 / §5 / §9 初心者ミス |
| デッキ思想・対面別戦略を `decks/*.json` のメタ情報に追加する | §11 AI研究環境 |

> **重要**: ここに書かれた「コツ」は人間の判断指針。engine 実装ヒューリスティックに落とすときは
> 数値閾値で近似する (例: 「無意味な攻撃をしない」→ `attacker.power < target.power` の枝刈り)。

## 1. 基本戦略

* 「**盤面**」と「**ライフ**」のどちらを優先するか常に考える
* 相手リーダーの色を見てゲームスピードを予測する
  * 赤系: アグロ寄り、ライフ詰めゲー
  * 青系: コントロール、終盤強い
  * 緑系: テンポ、レスト効果
  * 黒系: 盤面コントロール
  * 紫系: DON加速ランプ
  * 黄系: ライフ操作トリガー
* 勝ち筋を決めてからプレイする (リーサル○ターン目想定)
* 毎ターン「次ターン何をしたいか」を考える
* DONを余らせすぎない、ただしカウンター用に手札を減らしすぎない
* "今強い"より"次ターン強い"を意識する
* **無意味な攻撃をしない** (= パワー不足の確定失敗)
* 無理に全処理を狙わない
* 盤面有利でもリーサルを逃さない

> engine ヒューリスティック対応: `GreedyAI.choose_action` で
> `attacker.power < target.power` の攻撃を除外、`atk.power < opp.leader.power` の
> リーダー攻撃も除外。`engine/ai.py` の §2c 周辺を参照。

## 2. DON 管理

* DONは「今のターン」だけでなく「次ターン」の計画で使う
* 大型を出すターンを逆算する (例: 7コスト着地は turn 4-5)
* DONをリーダーに付けるか盤面に使うか毎回判断する
* **1DON 余らせることでイベントを構えられる** (相手ターン中の【カウンター】イベント発動コスト)
* 中途半端なDON配分は弱い (例: 5キャラに1ドンずつ配ると全部弱体化)
* 「**5000ライン**」「**7000ライン**」を意識する
  * リーダーは大体 5000 → +1000 で 6000 → 1k カウンター 1 枚で守れる
  * 7000 攻撃要求は 2k カウンター 1 枚 + 1k カウンター 1 枚 = 計 2 枚要求 (強い)
* DONを使い切ることが必ずしも正解ではない
* DON加速デッキ (紫系) はテンポ差を押し付ける
* 相手ターン中、自リーダー/キャラの DON は **+1000 効果なし** (公式 6-5-5、自ターン中のみ)

> engine 実装注意: `InPlay.is_owners_turn` フラグで DON+1000 ゲート済み (engine/core.py)。
> AI が DON を「防御用」に積むのは **無意味** (次の自リフレッシュで戻る)。
> AI ヒューリスティックでは DON は **攻撃打点を伸ばす目的** だけに使う前提でよい。

## 3. アタックの考え方

### アタック順

* **アタック順は超重要**
* 基本は「**小さい打点 → 大きい打点**」
  * 理由: 大きい打点を先に切ると、防御側がカウンター集中で守りやすい
  * 小さい打点を先に出すと、防御側はカウンターで守るかライフで受けるかの判断を強いられる
* 相手の手札枚数を意識する (= カウンター総量の上限)
* **7000 要求はかなり強い** (2 枚カウンター要求になりがち)

### 何を攻撃するか

* 無理にライフを削らない場面もある (= トリガーで返されるリスク)
* 相手に強いトリガーを引かせない判断も重要 (相手の残ライフ枚数 × トリガー率を計算)
* キャラ処理優先か顔優先かを決める
* リーダー効果起動のために先に攻撃することもある (例: 自リーダーをアクティブで残す要件)
* ブロッカーをどのタイミングで倒すか重要
  * ブロッカーを残すと毎ターン攻撃を吸われる
  * ただしブロッカー除去にコスト払うなら、その分テンポロス
* 相手のカウンター要求値を考えて殴る

### キャラ KO の優先度

* コスト 4 以上 = 高価値ターゲット (KO で大きなテンポ取れる)
* ブロッカー持ち = 優先 (毎ターン仕事するので長期的に重い)
* リーダー効果に関係するキャラ = 優先 (例: 「○○の特徴があれば〜」のシナジー要員)

> engine ヒューリスティック対応: `GreedyAI.choose_action` の 2a (キャラ KO 狙い) で
> `(target.cost, target.power)` でソート → 高コスト優先。アタック順は legal_actions の
> 順序に依存しているが、本来は「弱→強」順で出すべき。改善余地あり。

## 4. ライフ管理

* ライフは「HP」だけではなく「**リソース**」(受けると手札が増える)
* **早くライフを受ける方が強い場合もある** (序盤のライフは安い、手札に変換できる)
* 受けすぎるとリーサル圏に入る (= ライフ 1〜2 になると致命)
* **ライフ枚数より手札枚数が重要な場合がある**
* 終盤はライフ 1 枚の価値が非常に高い
* トリガー期待で受ける選択もある (リーダーのライフは内容を確認できる、デッキにトリガー多い色なら受け推奨)
* 無駄に守りすぎると手札が枯れる
* 相手のリーサル打点を毎ターン計算する

### ライフ別防御閾値の目安

| 残ライフ | カウンター切り判断 |
|---|---|
| 0 | 既に敗北 (リーダーが受ける状態 = ライフ 0 なら次ヒットで負け) |
| 1 | **致命**。手札全部使ってでも守る (counter_total 制限なし、3+ 枚も許容) |
| 2 | 致命予備軍。+8000 まで、3 枚まで許容 |
| 3 | 中盤。+6000 まで、2 枚まで |
| 4+ | 余裕。1 枚 (≤2000) のみ防御、それ以上は通す (= ライフを手札に変換) |

> engine ヒューリスティック対応: `GreedyAI.choose_defense` のライフ別閾値と一致。
> 上の表は engine の閾値を直接反映している。

## 5. カウンター管理

* **カウンター値は超重要**
* **2000 カウンターの枚数を意識する** (デッキ全体の防御力に直結)
* カウンターを吐きすぎると後半弱い
* 「**守るべきキャラ**」を決める (高コスト・キーカードを守る、低コストは見捨てる)
* リーダーで受けるかキャラを守るか判断する
* 相手の残 DON からイベントを警戒する (相手ターン中の【カウンター】イベント)
* **1 枚で守れるラインを意識する**
  * 1k カウンター 1 枚 = +1000 → ガップ ≤ 1000 の攻撃に対して
  * 2k カウンター 1 枚 = +2000 → ガップ ≤ 2000
  * 1k+2k = +3000 → ガップ ≤ 3000 (2 枚要求)
* 複数枚要求される攻撃 = 強い
* 相手の手札枚数からカウンター量を推測する

### キャラ攻撃へのカウンター

* 基本は **キャラ攻撃にはカウンター切らない** (リソース温存)
* 例外: コスト 4 以上の高価値キャラを 1 枚 (≤2000) で守る場合は OK

> engine ヒューリスティック対応: `GreedyAI.choose_defense` 末尾。

---

## 6. 盤面管理

* 盤面 5 面を埋める意味を考える
  * 攻撃手数が増える (アタッカー数 × 攻撃機会)
  * 全除去ケアの面ができる
* 無意味な横展開は **全除去に弱い** (赤紫ロジャー/エース系の全体除去カードに刈られる)
* **「次ターン生き残るキャラ」を意識する** (= パワー高い・KO 耐性ある・除去外しコスト)
* 除去されても役割を果たすキャラを使う (登場時効果で仕事する → KO でロスゼロ)
* アクティブキャラを残す意味を考える
  * アクティブ = 相手から攻撃されない (公式 7-1-1-2: アクティブ攻撃不可)
  * ただし「アクティブアタック可」キーワード持ち相手には逆効果
* ブロッカーを雑に切らない (毎ターン仕事するので長期的に重要)
* 低コストキャラをいつ切るか重要 (ブロッカー or アタッカー or サクリファイス)
* 相手の除去範囲を把握する (例: 相手のコスト 4 以下 KO に対して 5000 以上のキャラを並べる)
* 盤面テンポを取ると有利になりやすい

## 7. デッキ構築のコツ

* **「使いたいカード」から構築を始める** (このプロジェクトの目的そのもの)
* 相性カードを増やす (シナジー軸を太く)
* サーチカードは重要 (再現性 = 引きたい時に引ける)
* マナカーブ (コスト帯) を意識する
  * 1〜3 コスト: 序盤展開
  * 4〜5 コスト: 中盤の主力
  * 6+ コスト: フィニッシャー
* **カウンター値の総量を見る** (50 枚デッキで 30k 〜 50k が標準)
* 理想初動を決める
* 強い動きを再現できる枚数にする (= 4 枚積み)
* 引きたいカードは 4 枚寄り
* ピン刺しは明確な理由を持つ (例: 禁止リストで制限カード)
* 環境に刺さるカードを入れる (メタカード)
* 実際に回して調整する (このプロジェクトの matchup matrix で確認)

> engine 実装対応: `engine/deckbuilder.py` (コアカード固定型自動構築)、
> `/decks/[slug]/analyze` (色配分・コストカーブ・カウンター分布の可視化)。

## 8. 環境読み

* 「**今何が多いか**」を知る (cardrush.media や大会レポートを参照)
* Tier1 だけ見ない (Tier2-3 にもメタ刺さるデッキあり)
* 苦手対面を切る構築もある (= 5 マッチ中 1 つ捨てる代わりに他 4 つ厚く見る)
* 特定対面を厚く見ると他対面が弱くなる (トレードオフ)
* メタカードを入れすぎない (純度低下 = 本筋の動きが弱くなる)
* 流行カードを常に確認する
* **新弾後は環境が大きく変わる**
* 大会環境とローカル環境は違う

> このプロジェクトでは月次で `scripts/check_official_updates.py` + `scrape_cardrush_decks.py`
> でデッキ DB を更新。`scripts/select_cardrush_representatives.py` で代表選出。

## 9. 初心者がやりがちなミス (= AI に避けさせるべき行動パターン)

| ミス | 対応する engine ヒューリスティック |
|---|---|
| 全部守ろうとする | `choose_defense` ライフ ≥ 4 で過剰防衛しない |
| DON を無計画に使う | DON は攻撃打点が gap 内のときのみ付与 |
| キャラを出しすぎる | (現実装は出せるだけ出す。改善余地: 5 面埋まりに近づくと選別) |
| 顔を詰めるだけになる | キャラ KO を優先 (`choose_action` 2a) |
| **リーサル計算をしない** | 未実装。改善: 自分の合計打点が opp.life + opp.leader.power を超えるなら全攻撃 |
| カウンターを抱えすぎる | `choose_defense` で life=2-3 に counter_total/枚数を許容 |
| **攻撃順を考えない** | 未実装。改善: 「弱→強」順で攻撃 |
| 次ターンを考えずに動く | LookaheadAI が部分的にカバー (1-ply) |
| 相手の手札枚数を見ない | 未実装。改善: 相手の手札 0 枚なら強気攻撃 |
| 「強いカード」だけ入れてバランス崩壊 | デッキビルダー検証 (色・コスト・カウンター) |

## 10. 上級者ほど意識してること (= LookaheadAI 拡張の方向性)

* 相手の手札レンジ予測 (= ベイズ推論的な隠匿情報処理)
* 次ターンの最大値予測 (= 2-ply 探索)
* 残カウンター推定 (= プレイ済みカードからの逆算)
* 相手デッキタイプ推測 (= リーダー + 序盤プレイから絞り込み)
* "**何をケアするか**"の選択 (= 期待損失最小化)
* プレイの再現性 (= 分散の最小化)
* 勝ち筋の押し付け (= 相手の対応リソースを枯渇させる)
* 負け筋の回避 (= 致命プレイを除外)
* 相手の理想展開を崩す
* "今の 1 点"より"2 ターン後"

> engine 実装方向性: 現状の LookaheadAI は 1-ply。これらを実装するには
> 2-ply 以上 + 相手モデル + 隠匿情報の確率分布 が必要。Phase 7 以降の課題。

## 11. AI 研究環境で重要になる要素 (= デッキ JSON のメタ情報候補)

各メタデッキ JSON (`decks/cardrush_*.json`) に以下を付与すると AI が対面別戦略を取れる:

```json
{
  "leader": "OP01-001",
  "main": [...],
  "strategy": {
    "ideal_opening": ["OP01-013", "OP01-016"],   // 理想初動
    "deck_concept": "アグロ。3 ターン目までにライフ 2 詰める",
    "matchup_strategies": {
      "vs_aggro": "5000 ラインを切らさない、序盤からブロッカー展開",
      "vs_control": "ライフ受けで手札稼ぎ、長期戦に持ち込む"
    },
    "attack_priority": ["opp_leader_first", "ko_blocker", "ko_threat"],
    "lethal_priority": "high",   // リーサル可能なら他全部捨てる
    "guard_targets": ["OP01-016", "OP02-013"],   // 守るべきキャラ ID
    "mulligan_keep": ["OP01-013"],   // キープ基準
    "win_conditions": ["face_damage_lethal", "deck_out"],
    "lose_conditions": ["board_wipe_at_low_life", "ramp_outpaced"],
    "card_roles": {
      "OP01-013": "early_blocker",
      "OP02-013": "removal",
      "OP01-016": "search"
    }
  }
}
```

これらは現在 (2026-05) 未実装。AI が対面別に異なるヒューリスティックを取るには
これらのメタ情報を `engine/ai.py` の `choose_action` に渡す必要がある。
